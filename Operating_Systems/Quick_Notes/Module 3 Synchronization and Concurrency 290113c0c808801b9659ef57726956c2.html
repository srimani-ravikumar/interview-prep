Module 3: Synchronization and Concurrency

1. Process Synchronization 🔄

Definition:

Process synchronization ensures multiple processes can execute concurrently without interfering with each other or corrupting shared resources (critical section).

Real-Life Analogy:

Traffic management at a four-way intersection – traffic signals prevent collisions by controlling flow.

Need for Synchronization:

  • ✅ Shared Resource Management: Ensures safe access to memory, files, devices.
  • ✅ Prevent Race Conditions: Avoids conflicts from simultaneous access.
  • ✅ Data Consistency: Ensures predictable updates to shared data.
  • ✅ Deadlock & Starvation Avoidance: Prevents processes from indefinite waiting.

Tools for Synchronization:

  • Semaphores: Integer counters controlling resource access (wait(), signal()).
    • Binary (0 or 1) and Counting (range >1).
  • Mutexes: Locks for mutual exclusion in critical sections.
  • Monitors: High-level construct combining shared resources & code.
  • Barriers: Pause processes until all reach a point (used in parallel computing).

Challenges: Deadlocks, starvation, race conditions, priority inversion.

Advantages: Prevent conflicts, improve resource utilization, enable concurrency.

Disadvantages: Overhead, complexity, potential deadlocks.

Example Problems: Producer-Consumer, Readers-Writers, Dining Philosophers.

Takeaway: Proper synchronization is crucial for data integrity and stable concurrent systems.

Interview Focus 🔍: Semaphores vs Mutex, classic synchronization problems, race conditions.

2. Critical Section Problem 🛡️

Definition:

Occurs when multiple processes attempt to access shared resources simultaneously, risking inconsistency.

Real-Life Analogy:

Bank teller counter – only one customer can transact at a time.

Components:

  1. Entry Section: Requests access.
  1. Critical Section: Accesses shared resource.
  1. Exit Section: Leaves critical section.
  1. Remainder Section: Outside critical section.

Requirements for Solution:

  • Mutual Exclusion: Only one process in CS.
  • Progress: Processes decide who enters next.
  • Bounded Waiting: No indefinite waiting.

Solution Approaches:

  • Software: Peterson’s, Dekker’s Algorithm.
  • Hardware: TestAndSet, CompareAndSwap instructions.
  • OS-level: Semaphores, Mutexes, Monitors.

Advantages: Data integrity, race condition prevention, reliability.

Challenges: Efficiency, complexity, handling failures.

Examples: DBMS, Printer spooler, multithreaded apps.

Takeaway: Solving CS Problem ensures safe concurrent execution.

Interview Focus 🔍: Mutual exclusion, bounded waiting, progress property, Peterson’s algorithm.

3. Semaphores ⚙️

Definition:

Semaphore = integer counter managing concurrent process access to resources.

Real-Life Analogy:

Single-lane bridge with traffic lights controlling flow.

Types:

  • Counting Semaphore: Manages multiple identical resources.
  • Binary Semaphore: Acts as lock for single resource.

Operations:

  • wait(P): Decrements value; waits if 0.
  • signal(V): Increments value; signals resource release.

Advantages: Prevent race conditions, safe resource use, IPC facilitation.

Disadvantages: Deadlocks if misused, complex handling, performance overhead.

Examples: Producer-Consumer, Readers-Writers, Traffic light control.

Takeaway: Semaphores are versatile tools for process synchronization.

Interview Focus 🔍: Counting vs Binary semaphore, wait/signal operations, deadlock scenarios.

4. Producer-Consumer Problem 🍵

Definition:

Classic synchronization problem using a bounded buffer for producers (generate data) and consumers (process data).

Challenges:

  • Prevent buffer overflow (producer) and underflow (consumer).
  • Ensure mutual exclusion and coordination.

Real-Life Analogies:

  • Washroom with one key (mutual exclusion).
  • Coffee machine (producer) & customer (consumer).

Solution (Java example):

  • wait() → pause thread, release lock.
  • notify() / notifyAll() → resume waiting threads.
  • Bounded queue handles fast producers and slow consumers.

Key Points:

  • Single-slot buffer → prevents over-production.
  • Buffered queue → supports multiple submissions like TUF+ judge system.
  • Always use while loops with wait() to handle spurious wake-ups.

Takeaway: Thread coordination using wait-notify ensures safe and efficient communication.

Interview Focus 🔍: wait(), notify(), notifyAll(), bounded buffer implementation, handling fast producer/slow consumer.

5. Deadlock ⛔

Definition:

Deadlock = processes cannot proceed because each waits for a resource held by the other.

Real-Life Analogy:

Two drivers on a single-lane bridge waiting for the other to move.

Necessary Conditions:

  1. Mutual Exclusion
  1. Hold & Wait
  1. No Preemption
  1. Circular Wait

Handling Deadlock:

  • Prevention: Ensure at least one condition cannot occur.
  • Avoidance: Dynamically check for safe states (Banker’s Algorithm).
  • Detection & Recovery: Detect deadlock; terminate or preempt resources.

Banker’s Algorithm:

  • Simulates allocation to ensure safe state.
  • Steps: Resource request → safety check → allocate if safe.
  • Pros: Avoids unsafe states, maximizes utilization.
  • Cons: Needs max resource knowledge, high computation in large systems.

Takeaway: Deadlocks require careful handling via prevention, avoidance, or detection.

Interview Focus 🔍: Banker's Algorithm, safe state, deadlock conditions, prevention vs avoidance.

6. Starvation and Aging ⏳

Definition:

  • Starvation: Process waits indefinitely due to low priority.
  • Aging: Gradually increases waiting process priority to prevent starvation.

Real-Life Analogy:

Shy customer in grocery line → Starvation; store moves them forward → Aging.

Causes of Starvation:

  • Strict priority scheduling
  • High-priority resource monopolization
  • Lack of fairness

Solutions:

  • Aging
  • Fair scheduling algorithms (Round Robin, Fair Share)
  • Limit resource holding time

Advantages of Aging:

  • Ensures fairness
  • Improves system responsiveness
  • Compatible with multiple scheduling algorithms

Examples:

  • Disk scheduling (SCAN, C-SCAN)
  • Priority-based CPU scheduling

Takeaway: Aging prevents starvation and ensures fairness in resource allocation.

Interview Focus 🔍: Starvation scenarios, aging implementation, priority scheduling fairness.

Overall Key Takeaways ✅

  • Synchronization ensures data consistency, prevents race conditions, and avoids deadlocks.
  • Critical Section, Semaphores, Mutexes, Monitors, and wait-notify are essential tools.
  • Producer-Consumer problem models real-world concurrent coordination.
  • Deadlocks are prevented, avoided, or detected using Banker’s Algorithm and resource allocation strategies.
  • Starvation can be mitigated using aging to ensure fairness.
