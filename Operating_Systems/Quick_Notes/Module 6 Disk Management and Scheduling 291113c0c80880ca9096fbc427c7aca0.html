Module 6: Disk Management and Scheduling

Disk Management and Scheduling ensure efficient data access, optimized seek time, and fair I/O handling in operating systems.

Think of it as the OS being the traffic controller for disk requests 🚦— deciding who gets served first and how efficiently.

⚙️ 1. Disk Structure and Management

🧩 Disk Components

  • Platter: Circular disk where data is stored magnetically.
  • Track: Concentric circles on the platter.
  • Sector: Subdivision of a track (smallest addressable unit).
  • Cylinder: Stack of tracks aligned vertically across platters.
  • Disk Arm (Actuator): Moves read/write head to the required track.
  • Head: Reads/writes data on the disk surface.

📊 Diagram (conceptually)

Platter → Tracks → Sectors
Multiple Platters → Cylinders

📀 Disk Access Time

  1. Seek Time → Time to move head to desired track.
  1. Rotational Latency → Time for the desired sector to rotate under the head.
  1. Transfer Time → Time to read/write actual data.

⏱ Total Access Time = Seek Time + Rotational Latency + Transfer Time

🧰 2. Disk Management Functions

🔹 Disk Initialization

  • Partitioning and formatting before use.
  • Formatting includes creation of file system structures (boot block, superblock, inode table).

🔹 Boot Block

  • Contains bootloader to start OS.
  • Located at a fixed location on disk.

🔹 Bad Block Management

  • Some sectors may become unreadable.
  • OS remaps or marks them as bad.
  • Modern disks use spare sectors for replacement automatically.

💡 Takeaway: Disk management ensures reliability and prepares disks for efficient storage.

🚦 3. Disk Scheduling Algorithms

Efficient scheduling = minimizing seek time and improving throughput.

📚 Real-life analogy:

A librarian fetching books (requests) from shelves (tracks). Smart route = less walking (seek time).

🔸 1. First-Come, First-Served (FCFS)

  • Serves requests in the order of arrival.
  • No reordering or optimization.

📖 Example:

Requests: 40, 180, 10 → Serve sequentially.

✅ Advantages:

  • Simple, fair.

❌ Disadvantages:

  • High total seek time.
  • Poor performance when requests are far apart.

💬 Takeaway: Easy but inefficient.

🔍 Interview Tip: Often used as a baseline for comparison.

🔸 2. Shortest Seek Time First (SSTF)

  • Chooses nearest request from current head position.

📖 Example:

If head at 50 → Requests (10, 45, 80) → pick 45 first.

✅ Advantages:

  • Lower total seek time.
  • Better throughput.

❌ Disadvantages:

  • Starvation possible for far requests.
  • Unfair in heavy load.

💬 Takeaway: Fast but not fair.

🔍 Interview Tip: “How does SSTF cause starvation?” — explain head keeps choosing nearby requests.

🔸 3. SCAN (Elevator Algorithm)

  • Head moves in one direction, serving all requests.
  • At end → reverses direction like an elevator.

📈 Movement Example:

If head moves 20 → 80: serves requests in between; at 80, reverses.

✅ Advantages:

  • Uniform wait times.
  • Prevents starvation.

❌ Disadvantages:

  • Requests just missed at turnaround wait longer.

💬 Takeaway: Balanced algorithm for fairness and efficiency.

🔍 Interview Tip: Compare SCAN vs SSTF — trade-off between performance and fairness.

🔸 4. C-SCAN (Circular SCAN)

  • Moves in one direction only (e.g., outer → inner).
  • At end → jumps back to start without serving on return.

📈 Example:

20 → 80 (serve requests), jump to 0, continue forward.

✅ Advantages:

  • Predictable wait times.
  • Uniform response distribution.

❌ Disadvantages:

  • Idle movement when jumping back.

💬 Takeaway: SCAN with circular efficiency.

🔍 Interview Tip: "Why use C-SCAN over SCAN?" — better for uniform service time under heavy load.

🔸 5. LOOK and C-LOOK

  • Optimized versions of SCAN and C-SCAN.

📖 LOOK: Moves only up to last request in the direction → reverses.

📖 C-LOOK: Moves in one direction → stops at last request → jumps back.

✅ Advantages:

  • Avoids unnecessary traversal to empty ends.
  • Reduces total seek time.

❌ Disadvantages:

  • Slightly more complex implementation.

💬 Takeaway: Smarter SCAN and C-SCAN variants.

🔍 Interview Tip: Explain the "LOOK" name — the head looks ahead for farthest request.

⚙️ 6. Performance Metrics

  • Average Seek Time (AST) → Key metric to minimize.
  • Throughput → Requests served per unit time.
  • Fairness → Avoid starvation.
  • Response Time → Time between request submission and completion.

✨ Final Takeaways

  • Disk scheduling = optimizing movement for speed & fairness.
  • SCAN family (SCAN, C-SCAN, LOOK, C-LOOK) prevents starvation efficiently.
  • SSTF reduces seek time but may starve far requests.
  • Modern SSDs rely less on these but OS-level I/O schedulers still use similar logic.
