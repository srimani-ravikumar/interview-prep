1. System Calls and Their Categories üñ•Ô∏è

Definition:

A system call is a controlled interface provided by the OS for user applications to request services or resources from the kernel. User programs cannot access hardware directly, so system calls act as a secure bridge.

Real-Life Analogy:

A customer asking a service representative to perform tasks on behalf of the company (kernel) safely.

How System Calls Work:

  • User ‚Üí Kernel Transition: Switches from user mode to privileged kernel mode.
  • Request Handling: OS verifies parameters and permissions.
  • Result Delivery: Returns status or result to user application.

Categories of System Calls:

  1. Process Control: fork(), exec(), wait()
  1. File Management: open(), read(), write(), close()
  1. Device Management: ioctl(), read()/write() on devices
  1. Information Maintenance: getpid(), gettimeofday()
  1. Communication: pipe(), socket(), send(), recv()

Significance:

  • Security & stability
  • Hardware abstraction
  • Controlled resource management

Drawbacks:

  • Mode switching overhead
  • Complexity in usage
  • OS-specific compatibility

Takeaway: System calls are the backbone of OS-mediated access to hardware and resources.

2. System Boot Process ‚ö°

Definition:

Sequence from powering on a system to being fully operational.

Real-Life Analogy:

Restaurant setup: staff check tools (hardware), chef sets menu (bootloader), services ready to serve customers (OS and services).

Key Stages:

  1. POST: Hardware check (CPU, RAM, storage)
  1. BIOS/UEFI Initialization: Identify bootable devices
  1. Bootloader Execution: Load OS kernel (GRUB, Windows Boot Manager)
  1. Kernel Loading & Initialization: Resource management, device drivers setup
  1. Init/System Services Startup: Start essential services (systemd, daemons), show login/desktop

Legacy BIOS vs UEFI:

  • BIOS: Fixed memory, MBR-only, limited hardware init
  • UEFI: Modular, GPT support, secure boot, richer interface

Takeaway: Understanding boot stages helps troubleshooting, optimization, and secure startup.

Interview Focus üîç: POST, bootloader, kernel init, differences between BIOS & UEFI.

3. Interrupts ‚è±Ô∏è

Definition:

Signals that temporarily halt CPU execution to handle urgent events.

Types:

  • Hardware Interrupts: Triggered by external devices (keyboard, network card)
  • Software Interrupts: Triggered by programs or exceptions (division by zero)

Working Mechanism:

  1. Interrupt occurs
  1. CPU saves state
  1. ISR (Interrupt Service Routine) executes
  1. CPU resumes main program

Advantages:

  • Responsive system
  • Efficient CPU utilization
  • Scalable with multiple devices

Drawbacks:

  • Context switching overhead
  • ISR complexity
  • Priority conflicts

Real-World Usage: Keyboard input, network packet arrival, system calls, error handling

Takeaway: Interrupts allow reactive and efficient OS behavior.

Interview Focus üîç: Hardware vs software interrupts, ISR mechanism, advantages/disadvantages.

4. Event-Driven Programming in OS üåÄ

Definition:

Flow of execution determined by external/internal events instead of sequential code.

Working:

  1. Event Registration: Handlers/callbacks linked to events
  1. Event Detection: OS monitors inputs, interrupts, queues
  1. Event Dispatching: Event triggers corresponding handler
  1. Event Handling: Handler executes, then returns control

Key Features:

  • Asynchronous execution
  • Event loops and handlers
  • Callback functions
  • Interrupt-driven processing

Advantages:

  • High responsiveness
  • Efficient resource use
  • Scalable and modular

Disadvantages:

  • Debugging complexity
  • Event overload risk
  • Steep learning curve

Real-World Usage: GUI apps, network servers, kernel interrupt handling

Takeaway: Event-driven OS design improves responsiveness and efficiency.

Interview Focus üîç: Event loops, callbacks, asynchronous handling, advantages/disadvantages.

6. Priority Inversion ‚ö†Ô∏è

Definition:

High-priority tasks blocked by lower-priority tasks holding shared resources.

Causes:

  • Non-preemptive resources
  • No priority inheritance
  • Long critical sections

Effects:

  • Delayed critical tasks
  • Inconsistent system behavior
  • Missed deadlines in real-time systems

Solutions:

  • Priority Inheritance: Temporarily boost low-priority task
  • Priority Ceiling Protocol: Resource-level priority ceiling
  • Well-defined locking strategies

Example: High-priority task waiting for sensor buffer held by low-priority task; medium-priority task preempts low-priority task.

Takeaway: Priority inversion can degrade system performance; mitigated via inheritance/ceiling protocols.

Interview Focus üîç: Causes, effects, and mitigation strategies.

7. Load Balancing in Operating Systems ‚öñÔ∏è

Definition:

Distributing workloads across processors/resources to optimize performance and avoid overload.

Types:

  • Static: Predefined allocation rules, simpler but less flexible
  • Dynamic: Runtime decisions based on current load

Common Algorithms:

  • Round-Robin
  • Least Connections
  • Weighted Distribution

Advantages:

  • Maximizes resource utilization
  • Reduces response time, increases throughput
  • Improves fault tolerance

Challenges:

  • Overhead in monitoring
  • Fairness in dynamic conditions
  • Heterogeneous system handling

Real-World Usage: Web servers, cloud platforms, data centers

Takeaway: Load balancing ensures efficient, stable, and fault-tolerant system operation.

Interview Focus üîç: Load balancing types, algorithms, advantages/challenges.
