Module 7: Inter-Process Communication (IPC)

Inter-Process Communication (IPC) enables processes to exchange data, synchronize actions, and share resources, whether on the same machine or across a network.

One-liner takeaway: IPC ensures processes communicate and coordinate efficiently like team members in a collaborative environment.

Interview Focus 🔍: Explain IPC mechanisms, differences between Message Passing & Shared Memory, and common use cases in OS design.

1. IPC Mechanisms Overview 📨

  • Definition: Mechanisms allowing processes to communicate and synchronize.
  • Purpose:
    • Share data efficiently.
    • Coordinate tasks.
    • Prevent resource conflicts.
  • Real-life Analogy: Team members exchanging notes or discussing tasks in a meeting room.
  • Key Mechanisms:
    1. Message Passing
    1. Shared Memory

2. Message Passing IPC ✉️

Definition: Processes communicate by sending/receiving structured messages through OS-managed channels.

Real-life Analogy: Two colleagues using a shared mailbox to exchange notes without accessing each other’s desks.

Working Steps:

  1. Message Creation: Sender structures data.
  1. Sending: OS manages delivery via message queues/pipes.
  1. Receiving: Receiver reads messages (blocking or non-blocking).
  1. Processing: Receiver acts on message; optionally sends a response.

Advantages:

  • Isolation: Processes remain independent.
  • Modularity: Well-defined communication channels.
  • Synchronization: Send/receive can synchronize actions.
  • Network-friendly: Works across distributed systems.

Disadvantages:

  • Overhead: Message copying and context switches.
  • Complexity: Ordering and delivery guarantees may require extra logic.
  • Buffering issues: Limited queue sizes can cause delays.

One-liner takeaway: Message passing provides safe, structured IPC ideal for distributed systems.

Interview Focus 🔍: Differences between synchronous/asynchronous message passing; use cases for distributed applications.

3. Shared Memory IPC 🖥️

Definition: Processes share a common memory region for direct data exchange.

Real-life Analogy: Team members using a whiteboard in a meeting room, coordinating via rules to avoid conflicts.

Working Steps:

  1. Memory Allocation: OS allocates shared segment.
  1. Attach: Processes attach to memory segment.
  1. Access: Read/write data directly.
  1. Synchronization: Use mutexes/semaphores to avoid conflicts.

Advantages:

  • High performance: Direct memory access.
  • Large data sharing without repeated copying.
  • Efficient communication once set up.

Disadvantages:

  • Complex synchronization (mutexes, semaphores required).
  • Memory management (allocation & cleanup).
  • Usually confined to a single machine.

One-liner takeaway: Shared memory is fastest for local high-volume IPC but requires careful synchronization.

Interview Focus 🔍: Difference between message passing and shared memory, pros/cons, synchronization challenges.

4. Key Differences: Message Passing vs Shared Memory 🔄

FeatureMessage PassingShared Memory
Data ExchangeDiscrete messages sent/receivedDirect memory access
SynchronizationImplicit via send/receiveExplicit via locks/semaphores
PerformanceModerate; kernel interventionHigh; direct access
Best UseDistributed systemsProcesses on same machine

5. Signals and Signal Handling 🔔

Definition: Asynchronous software interrupts sent by OS to notify processes of events.

Real-life Analogy: Reading a book at home and suddenly the doorbell rings; you decide how to respond.

Signal Sources:

  • User interaction: Ctrl+C → SIGINT
  • Software exceptions: Division by zero → SIGFPE
  • Kernel events: Timer expiry, hardware changes
  • Other processes: kill() system call

Signal Delivery & Default Actions:

  • Queued and delivered by OS.
  • Default actions:
    • SIGTERM: Terminate gracefully.
    • SIGKILL: Forced termination.
    • SIGSTOP: Pause execution.

Custom Signal Handlers:

  • Override default behavior with a function.
  • Graceful shutdown: Close files, free memory.
  • Ignore signals when appropriate.
  • Restart operations after handling.

Common Signals:

  • SIGINT: Ctrl+C, interrupt process.
  • SIGTERM: Polite termination request.
  • SIGKILL: Forceful termination (cannot be caught).
  • SIGSEGV: Segmentation fault (invalid memory access).
  • SIGALRM: Timer-triggered events.

Steps in Signal Handling:

  1. Signal raised.
  1. Delivered by OS.
  1. Handler executes (custom or default).
  1. Process resumes (or terminates).

Advantages:

  • Immediate notification of critical events.
  • Low overhead.
  • Flexible handling.

Challenges:

  • Asynchronous complexity.
  • Limited information in signal.
  • Safe handler design required.

One-liner takeaway: Signals provide lightweight, asynchronous process notifications for rapid OS-level event handling.

Interview Focus 🔍: Use cases of signals in UNIX/Linux, handling SIGTERM vs SIGKILL, and implementing signal handlers safely.

Summary:

IPC ensures processes communicate, synchronize, and share resources effectively. Message Passing and Shared Memory cover structured and high-speed local communication, while Signals provide asynchronous notifications. Choosing the right mechanism depends on system architecture, performance needs, and synchronization complexity.
