A. Language Fundamentals
1. Data Types

üìå What problem does this solve?

Understanding how data is stored, copied, and passed in C# is critical to:

  • Avoid unintended side effects
  • Write performant code
  • Correctly reason about method behavior

Most real interview bugs come from misunderstanding this topic.

üß† Intuition (Plain English)

‚ÄúDo I get a copy of the data, or a pointer to the same data?‚Äù

That single question defines value vs reference.

üß© How It Works Internally (Mental Model)

Value Type

  • Holds the actual data
  • Assignment ‚Üí copies data
  • Each variable is independent

Reference Type

  • Holds a reference (address)
  • Assignment ‚Üí copies reference
  • Multiple variables point to same object

üîπ Value Types

Characteristics

  • Stored directly (usually on stack)
  • Can move to heap via boxing
  • Fast allocation & cleanup
  • Copy semantics

Examples

int,float,double,bool,char,struct,enum

Behavior

int a =10;
int b = a;// COPY of value

b =20;

Console.WriteLine(a);// 10
Console.WriteLine(b);// 20

‚úî Changes do not affect each other.

üîπ Reference Types

Characteristics

  • Object lives on heap
  • Stack stores reference
  • Shared mutable state
  • Reference semantics

Examples

class,interface,array,string,object,delegate,record

Behavior

Person p1 =new Person();
Person p2 = p1;// COPY of reference

p2.Age =30;

Console.WriteLine(p1.Age);// 30

‚úî Both point to same object.

‚≠ê WHY VALUE VS REFERENCE MATTERS (Interview Gold)

1Ô∏è‚É£ Performance

  • Value types ‚Üí cheap allocation, no GC pressure
  • Reference types ‚Üí heap allocation + GC

2Ô∏è‚É£ Mutability

  • Value types ‚Üí isolated changes
  • Reference types ‚Üí shared changes

3Ô∏è‚É£ Method Passing Semantics

  • Value type ‚Üí copy passed
  • Reference type ‚Üí reference copied

‚ö† Reference is copied, not the object

üß© ONE Consolidated Code Example

using System;

class Program
{
    static void Main()
    {
        // -------- VALUE TYPE --------
        int x = 10;
        ModifyValue(x);
        Console.WriteLine(x); // 10 (unchanged)

        // -------- REFERENCE TYPE --------
        Person p = new Person { Age = 25 };
        ModifyReference(p);
        Console.WriteLine(p.Age); // 99 (changed)
    }

    static void ModifyValue(int value)
    {
        value = 99;
        // Modifies local COPY only
    }

    static void ModifyReference(Person person)
    {
        person.Age = 99;
        // Modifies the SAME object on heap
    }
}

class Person
{
    public int Age { get; set; }
}

2. Type Conversions

üìå What problem does this solve?

Different data types cannot always interact directly.

Type conversions allow you to:

  • Move data between types safely
  • Handle user input and external data
  • Control precision and data loss explicitly

üß† Intuition (Plain English)

‚ÄúCan this value fit into the target type without losing information?‚Äù
  • If yes ‚Üí implicit
  • If no ‚Üí explicit or conversion API

üß© How Type Conversion Works Internally

  • Implicit conversions are checked at compile time
  • Explicit conversions may cause data loss
  • Convert and Parse perform runtime checks
  • Boxing may occur when converting to object

üîç Types of Conversions

1Ô∏è‚É£ Implicit Conversion

  • Safe
  • No data loss
  • No cast required
int x =10;
long y = x;// int ‚Üí long

‚úî Widening numeric conversions only

‚ùå No implicit conversion from long ‚Üí int

2Ô∏è‚É£ Explicit Conversion (Casting)

  • Risky
  • Data loss possible
  • Cast required
double d =10.5;
int i = (int)d;// truncates ‚Üí 10

‚ö† Edge cases (important):

double big =1e20;
int x = (int)big;// overflow ‚Üí undefined value


int a =int.MaxValue;
checked
{
    a++;// throws OverflowException
}

üí° Interview tip:

Casting does not round ‚Äî it truncates

3Ô∏è‚É£ Convert Class

  • Handles multiple input types
  • Safer for runtime data
  • Converts null ‚Üí 0
Convert.ToInt32(null);// 0
Convert.ToInt32("123");// 123

‚ö† Edge cases:

Convert.ToInt32("10.5");// FormatException
Convert.ToInt32(true);// 1
Convert.ToInt32(false);// 0

üí° Uses IConvertible internally

4Ô∏è‚É£ Parse vs TryParse

Used mainly for string ‚Üí value type conversions.

int.Parse("123");// OK
int.TryParse("123",outint n);// true

‚ö† Edge cases:

int.Parse(null);// ArgumentNullException
int.Parse("abc");// FormatException


bool ok =int.TryParse(null,outint x);// false

‚úî TryParse never throws

‚úî Always prefer for user input / APIs

‚ùó Common Interview Traps

  1. ‚ùå ‚ÄúCasting rounds values‚Äù

    ‚Üí Casting truncates

  1. ‚ùå ‚ÄúConvert never throws‚Äù

    ‚Üí Throws on invalid formats

  1. ‚ùå ‚ÄúTryParse gives default value‚Äù

    ‚Üí It returns false, output is default

  1. ‚ùå ‚ÄúOverflow always throws‚Äù

    ‚Üí Only inside checked

    static void Main()
    {
        // -------- IMPLICIT --------
        int a = 10;
        long b = a;

        // -------- EXPLICIT --------

        int x = Convert.ToInt32(null);     // 0
        int y = Convert.ToInt32("123");    // 123

        int p = int.Parse("456");

            checked

            }

3. var vs dynamic

üìå What problem does this solve?

Both var and dynamic let you avoid explicitly writing a type name, but they solve very different problems:

  • var ‚Üí reduces verbosity
  • dynamic ‚Üí enables runtime flexibility

Misunderstanding this causes runtime crashes and bad design choices.

üß† Intuition (Plain English)

var: ‚ÄúCompiler, you figure out the type ‚Äî I still want safety.‚Äù

dynamic: ‚ÄúI‚Äôll figure it out at runtime ‚Äî trust me.‚Äù

üîç var

Characteristics

  • Compile-time typing
  • Type is fixed at compile time
  • Full IntelliSense & refactoring support
  • Zero runtime cost
var x =10;// int
var s ="text";// string

‚ùå Cannot change type later:

var x =10;
x ="hello";// compile-time error

‚ö† Edge cases (important):

var n =null;// ‚ùå compile-time error (type cannot be inferred)


var list =new[] {1,2,3 };// int[]

üí° Interview truth:

var is not dynamic ‚Äî it‚Äôs just syntactic sugar.

üîç dynamic

Characteristics

  • Runtime typing (late binding)
  • No compile-time checks
  • No IntelliSense safety
  • Errors surface only at runtime
dynamic d =10;
d ="hello";// allowed


dynamic d =10;
d.Fly();// RuntimeBinderException

‚ö† Edge cases (very important):

dynamic x =null;
x.ToString();// RuntimeBinderException (null reference)


dynamic obj =new { Name ="John" };
Console.WriteLine(obj.Name);// works
Console.WriteLine(obj.Age);// runtime error

üí° Uses Dynamic Language Runtime (DLR) under the hood.

‚≠ê WHY THIS MATTERS (Interview Gold)

Performance

  • var ‚Üí compile-time resolved ‚Üí fast
  • dynamic ‚Üí runtime binder ‚Üí slower

Safety

  • var ‚Üí compile-time guarantees
  • dynamic ‚Üí runtime crashes possible

Usage intent

  • var ‚Üí readability & refactoring
  • dynamic ‚Üí interoperability & flexibility

‚ùó Common Interview Traps

  1. ‚ùå ‚Äúvar is weakly typed‚Äù

    ‚Üí No, it‚Äôs strongly typed

  1. ‚ùå ‚Äúdynamic skips type checking entirely‚Äù

    ‚Üí Checked at runtime, not skipped

  1. ‚ùå ‚Äúdynamic is only for JavaScript-like coding‚Äù

    ‚Üí Also used for COM, JSON, reflection-heavy APIs

üß© ONE Consolidated Code Example

        // -------- var --------

        var b = "hello";  // string

4. Nullable Types

üìå What problem does this solve?

Value types (int, bool, DateTime) cannot represent ‚Äúno value‚Äù.

Nullable types allow value types to:

  • Represent absence of data
  • Model optional fields
  • Avoid magic values like 1 or DateTime.MinValue

üß† Intuition (Plain English)

‚ÄúThis value type may or may not have a value.‚Äù

Instead of inventing fake defaults, nullable types make absence explicit.

üß© How Nullable Types Work Internally

Is syntax sugar for:

Nullable<int> n;

Internally it contains:

  • bool HasValue
  • int Value

‚ùó Common Interview Traps

  1. ‚ùå ‚ÄúNullable prevents null reference exceptions‚Äù

    ‚Üí Only if used correctly

  1. ‚ùå ‚ÄúNullable is same as reference null‚Äù

    ‚Üí Value types behave very differently

  1. ‚ùå ‚ÄúAccessing .Value is safe‚Äù

    ‚Üí Throws if HasValue == false

üß© ONE Consolidated Code Example

using System;

class Program
{
    static void Main()
    {
        // -------- NULLABLE VALUE TYPE --------
        int? n = null;

        // Safe check
        if (n.HasValue)
        {
            Console.WriteLine(n.Value);
        }

        // -------- NULL-COALESCING (??) --------
        int x = n ?? 0; // fallback if null

        // -------- NULL-COALESCING ASSIGNMENT (??=) --------
        n ??= 10; // assign only if n is null

        // -------- SAFE VALUE EXTRACTION --------
        int y = n.GetValueOrDefault();     // 10
        int z = n.GetValueOrDefault(99);   // custom default

        // -------- NULL-PROPAGATION (?.) --------
        Person person = null;
        string city = person?.Address?.City;
        // null-safe navigation

        // -------- NULLABLE BOXING --------
        int? a = 10;
        int? b = null;

        object boxedA = a; // boxes int
        object boxedB = b; // becomes null

        Console.WriteLine(boxedA);          // 10
        Console.WriteLine(boxedB == null);  // True
    }
}

{
    public Address Address { get; set; }
}

{

5. Strings

üìå What problem does this solve?

Strings represent textual data, but they are used everywhere ‚Äî logging, APIs, UI, file handling.

C# makes strings immutable by design to ensure:

  • Thread safety
  • Predictable behavior
  • Performance optimizations (interning)

üß† Intuition (Plain English)

‚ÄúA string never changes ‚Äî every ‚Äòchange‚Äô creates a new string.‚Äù

This design:

  • Prevents accidental modification
  • Enables memory sharing
  • Avoids subtle multi-threading bugs

üß© How Strings Work Internally

  • string is a reference type
  • Stored on the heap
  • Immutable
  • Backed by a character array
  • Literal strings are interned

üîç Core String Characteristics

FeatureBehavior
TypeReference type
Mutability‚ùå Immutable
Thread-safe‚úÖ Yes
EqualityValue-based
InterningAutomatic for literals

‚ùó Common Interview Traps

  1. ‚ùå ‚ÄúString is a value type‚Äù

    ‚Üí No, it‚Äôs a reference type with value semantics

  1. ‚ùå ‚Äú+= modifies the same string‚Äù

    ‚Üí It creates a new object

  1. ‚ùå ‚ÄúStringBuilder is always faster‚Äù

    ‚Üí Only for repeated modifications

üß© ONE Consolidated Code Example

using System;
using System.Text;

class Program
{
    static void Main()
    {
        // -------- IMMUTABILITY --------
        string a = "Hello";
        string b = a + " World"; // NEW string created

        Console.WriteLine(a); // Hello
        Console.WriteLine(b); // Hello World

        // -------- STRING INTERNING --------
        string s1 = "test";
        string s2 = "test";

        Console.WriteLine(object.ReferenceEquals(s1, s2)); // True (same memory)

        string s3 = new string("test".ToCharArray());
        Console.WriteLine(object.ReferenceEquals(s1, s3)); // False (not interned)

        // -------- BAD: STRING CONCATENATION IN LOOP --------
        string result = "";
        for (int i = 0; i < 3; i++)
        {
            result += i;
            // Each iteration:
            // 1. Creates a new string
            // 2. Copies old content
            // 3. Discards previous string (GC pressure)
        }

        // -------- GOOD: STRINGBUILDER --------
        var sb = new StringBuilder();
        for (int i = 0; i < 3; i++)
        {
            sb.Append(i); // Mutates internal buffer
        }

        string finalResult = sb.ToString();
        Console.WriteLine(finalResult);
    }
}

üèé StringBuilder (When & Why)

Use StringBuilder when:

  • Concatenating inside loops
  • Building large strings dynamically
  • Performance is critical

Avoid StringBuilder when:

  • Few concatenations
  • Readability matters more than micro-optimizations

üéØ Interview Question: Why is string concatenation inside loops bad?

‚ÄúBecause strings are immutable, every concatenation creates a new object, causing repeated allocations, copying, and GC overhead.‚Äù

8. Records

üìå What problem does this solve?

In real-world applications, we often need to model data, not behavior ‚Äî such as DTOs, API models, events, and messages.

Before records, using classes for this caused problems:

  • Equality compared references, not data
  • Required manual overrides of Equals() and GetHashCode()
  • Immutability had to be enforced manually
  • Easy to introduce bugs and boilerplate

Records solve this by:

  • Making value-based equality the default
  • Encouraging immutable data models
  • Eliminating repetitive boilerplate code
  • Making data objects safer and more expressive by design
In short: records make data correctness the default instead of an afterthought.

What is a record?

A record is a data-centric type (introduced in C# 9) designed to:

  • Represent immutable data
  • Use value-based equality
  • Reduce boilerplate code

It can be either:

  • record class ‚Üí reference type
  • record struct ‚Üí value type

Deep Dive Concept: Deferred Execution

This is the most critical concept in understanding LINQ's performance behavior.

Most LINQ methods do not execute the query immediately when you define it. They return a query object that defers execution until the results are actually needed (for example, when you loop through the results using foreach, call ToList(), ToArray(), Count(), or use one of the Element Operators).

This allows you to chain many operations efficiently.

// DEFERRED EXECUTION: The query is defined but not run yet.
var query = products.Where(p => p.Price > 50m);
Console.WriteLine("Query defined, no loop started yet.");

// EXECUTION TRIGGERS HERE: The results are pulled the first time we loop.
foreach (var p in query)
{
    Console.WriteLine($"Result 1: {p.Name}");
}

// EXECUTION TRIGGERS AGAIN HERE: The logic runs from the start a second time.
var count = query.Count();
Console.WriteLine($"Count: {count}");

Methods that trigger immediate execution are called Eager or Buffering methods (ToList(), ToArray(), Sum(), First(), etc.).

Methods that defer execution are called Lazy methods (Where(), Select(), OrderBy(), GroupBy()).

For extensive documentation and examples of all standard query operators, refer to the Microsoft Learn LINQ documentation.

    • = allowed (unsubscribe)
    • Direct invocation ‚ùå blocked from outside

Internally:

But exposed as:

Meaning:

  • OnClick ‚Üí subscriber method
  • += ‚Üí subscribe safely

External code:

  • ‚ùå Cannot invoke
  • ‚ùå Cannot overwrite
  • ‚úÖ Can only subscribe/unsubscribe

2. GC Generations

üìå What problem does this solve?

Garbage Collection (GC) exists to:

  • Automatically manage memory
  • Prevent memory leaks
  • Free developers from manual malloc/free

But scanning the entire heap every time is slow.

üëâ Generations solve this performance problem.

üß† Intuition (Plain English)

‚ÄúMost objects die young ‚Äî so clean young objects more often.‚Äù

Instead of scanning everything:

  • GC focuses first on short-lived objects
  • Older objects are assumed to be still useful

This dramatically improves performance.

üß© How GC Generations Work Internally

The managed heap is divided into generations:

GenerationDescription
Gen 0Newly allocated, short-lived objects
Gen 1Survivors of Gen 0
Gen 2Long-lived objects
LOHLarge Object Heap (special case)

Objects are promoted as they survive collections.

üîπ Generation 0 (Gen 0)

Characteristics

  • Holds new objects
  • Smallest generation
  • Collected very frequently
  • Fastest to clean
var obj = new object();// Gen 0

üß† Most allocations happen here.

üîπ Generation 1 (Gen 1)

Characteristics

  • Acts as a buffer between Gen 0 and Gen 2
  • Holds objects that survived one GC
  • Collected less frequently than Gen 0

üß† Helps prevent promoting short-lived objects too quickly.

üîπ Generation 2 (Gen 2)

Characteristics

  • Holds long-lived objects
  • Collected rarely
  • Most expensive to clean

Examples:

  • Caches
  • Static data
  • Singleton objects

üß† Gen 2 collection often causes noticeable pauses.

üîπ Large Object Heap (LOH) ‚Äî Interview Favorite

What qualifies?

  • Objects ‚â• 85 KB
  • Large arrays
  • Large strings
byte[] buffer = new byte[100_000];// LOH

Key Points

  • Allocated directly into LOH
  • Collected only during Gen 2 GC
  • Historically not compacted (fragmentation risk)

üß† Object Promotion Flow

Allocate ‚Üí Gen 0
Survive ‚Üí Gen 1
Survive again ‚Üí Gen 2

Once in Gen 2:

  • Object stays there until collected
  • Promotion is one-way

    {
        longLived = new object(); // Promoted to Gen 2
    }
}

üìå What problem does this solve?

It cannot automatically clean up unmanaged resources such as:

  • DB connections leak
  • System resources exhaust

üß† Intuition (Plain English)

‚ÄúThis object owns unmanaged resources and knows how to release them.‚Äù

}

  • Close database connections
  • Release sockets / handles
  • Free native resources
  • Make object unusable afterward

What using Guarantees

4. Finalizers

üìå What problem does this solve?

Finalizers exist to provide a last-chance cleanup mechanism for unmanaged resources if the developer forgets to call Dispose().

They are not meant for normal resource management.

üß† Intuition (Plain English)

‚ÄúA backup cleanup plan ‚Äî slow, unreliable, and should almost never be needed.‚Äù
  • Dispose() ‚Üí primary cleanup
  • Finalizer ‚Üí emergency fallback

üß© What Is a Finalizer?

A finalizer is a special method:

  • Defined using a destructor syntax
  • Executed by the GC, not the developer
  • Runs at an unknown time in the future
~MyClass()
{
		// Cleanup logic
}

üß© How Finalizers Work Internally

When an object has a finalizer:

  1. Object becomes unreachable
  1. GC does NOT immediately free memory
  1. Object is placed on finalization queue
  1. Finalizer thread executes ~MyClass()
  1. Object survives one extra GC cycle
  1. Memory is finally reclaimed later

‚ö†Ô∏è This makes finalizers expensive.

‚ùó Why Finalizers Slow Down GC

  • Objects with finalizers:
    • Live longer
    • Survive at least one GC
    • Cannot be reclaimed immediately
  • Finalizer thread runs serially

üß© ONE Consolidated Code Example

using System;

class ResourceHolder
{
    private bool _disposed;

    // Finalizer (destructor syntax)
    ~ResourceHolder()
    {
        Dispose(false);
    }

    public void Dispose()
    {
        Dispose(true);
        GC.SuppressFinalize(this); // Prevent finalizer
    }

    protected virtual void Dispose(bool disposing)
    {
        if (_disposed) return;

        if (disposing)
        {
            // Free managed resources
        }

        // Free unmanaged resources
        _disposed = true;
    }
}

üß† Intuition (Plain English)

üß© How Boxing & Unboxing Work Internally

4. CancellationToken

üìå What problem does this solve?

In .NET, tasks and async operations cannot be forcefully stopped safely.

  • Killing threads is dangerous
  • Forced termination can corrupt state
  • Resources may not be released

üëâ CancellationToken enables cooperative cancellation.

üß† Intuition (Plain English)

‚ÄúPlease stop when you reach a safe point.‚Äù
  • Caller requests cancellation
  • Callee decides when and how to stop
  • No force, no corruption

üß† Think of it as a polite stop signal, not a kill switch.

üß© Core Components

Cancellation in .NET has three parts:

ComponentRole
CancellationTokenSourceIssues cancellation
CancellationTokenPassed to methods
Consumer codeChecks token & stops

üß© Basic Cancellation Flow

CancellationToken token = cts.Token;

// Request cancellation
cts.Cancel();

üß© How CancellationToken Works Internally

  • CancellationToken is a struct
  • It references a shared CancellationTokenSource
  • IsCancellationRequested checks a flag
  • ThrowIfCancellationRequested() throws OperationCanceledException

‚ö†Ô∏è No thread is stopped automatically

üß© ONE Consolidated Code Example

using System;
using System.Threading;
using System.Threading.Tasks;

class Program
{
    static async Task Main()
    {
        var cts = new CancellationTokenSource();

        Task work = DoWorkAsync(cts.Token);

        // Cancel after 2 seconds
        await Task.Delay(2000);
        cts.Cancel();

        try
        {
            await work;
        }
        catch (OperationCanceledException)
        {
            Console.WriteLine("Work cancelled safely");
        }
    }

    static async Task DoWorkAsync(CancellationToken token)
    {
        for (int i = 0; i < 10; i++)
        {
            token.ThrowIfCancellationRequested();
            Console.WriteLine($"Working... {i}");
            await Task.Delay(500);
        }
    }
}

5. Deadlocks

üìå What problem does this solve?

Deadlocks occur when:

  • A thread is blocked
  • While the continuation needs that same thread
  • Result: no one can proceed

This is one of the most common real-world async bugs in C#.

üß† Intuition (Chef Analogy üç≥)

‚ÄúThe chef starts cooking rice, then sits down and waits,

but the timer alarm also needs the chef to respond.‚Äù

üß† Chef = Thread

üß† Rice = Async Task

üß† Timer callback = await continuation

‚û°Ô∏è Chef waits for rice

‚û°Ô∏è Rice completion waits for chef

‚û°Ô∏è Deadlock

üß© The Root Cause

‚ùå Blocking async code:

or

Why this deadlocks

  1. UI / ASP.NET thread calls async method
  1. Thread is blocked by .Wait() / .Result()
  1. Async method completes
  1. Continuation tries to resume on same thread
  1. Thread is blocked ‚Üí deadlock

üéØ Interview Gold Line

‚ÄúThe continuation needs the thread that is blocked.‚Äù
